# Stripe Local Testing — RetirePlanner.ca

After going live, `STRIPE_SECRET_KEY` in Supabase prod secrets is a live key.
Stripe will reject test price IDs against a live key, so local dev cannot call
the deployed edge functions directly for payment flows.

This document covers the correct local testing setup: run edge functions locally
with test secrets and forward test webhooks via the Stripe CLI.

---

## Prerequisites

### 1. Install the Stripe CLI

```bash
# macOS (Homebrew)
brew install stripe/stripe-cli/stripe

# Linux (Debian/Ubuntu)
curl -s https://packages.stripe.dev/api/deb/apt-transport-stripe/stripe-transport.gpg | sudo gpg --dearmor -o /usr/share/keyrings/stripe.gpg
echo "deb [signed-by=/usr/share/keyrings/stripe.gpg] https://packages.stripe.dev/api/deb stable main" | sudo tee /etc/apt/sources.list.d/stripe.list
sudo apt update && sudo apt install stripe

# Verify
stripe --version
```

Authenticate once:

```bash
stripe login
# Opens browser → authorize in Stripe Dashboard (test mode is fine)
```

### 2. Install Deno (required by Supabase functions serve)

```bash
curl -fsSL https://deno.land/install.sh | sh
# Add to PATH as directed by the installer
```

### 3. Install the Supabase CLI (if not already)

```bash
npm install -g supabase
supabase --version
```

---

## Step 1 — Create your local secrets file

Copy the example template and fill in your test values:

```bash
cp supabase/functions/.env.local.example supabase/functions/.env.local
```

Open `supabase/functions/.env.local` and fill in:

| Key | Where to find it |
|-----|-----------------|
| `STRIPE_SECRET_KEY` | Stripe Dashboard → Developers → API Keys → **Test** Secret key |
| `STRIPE_WEBHOOK_SECRET` | See Step 2 below — generated by `stripe listen` |
| `SUPABASE_URL` | Supabase Dashboard → Project Settings → API → Project URL |
| `SUPABASE_ANON_KEY` | Supabase Dashboard → Project Settings → API → anon public |
| `SUPABASE_SERVICE_ROLE_KEY` | Supabase Dashboard → Project Settings → API → service_role secret |
| `APP_URL` | `http://localhost:5173` for local dev |

> `supabase/functions/.env.local` is gitignored — never commit it.

---

## Step 2 — Get the test webhook signing secret

The webhook secret is generated per-session by the Stripe CLI listener.
Run this first to get the secret, then paste it into `.env.local`:

```bash
stripe listen --print-secret
# Prints: whsec_test_...
```

Copy that value into `STRIPE_WEBHOOK_SECRET` in `supabase/functions/.env.local`.

---

## Step 3 — Start the Supabase functions server

```bash
cd /DATA/retirement
supabase functions serve --env-file supabase/functions/.env.local
```

This starts all edge functions locally on port `54321` (or `54322` depending on
your Supabase version). The functions read secrets from `.env.local` instead of
the deployed Supabase secrets.

You should see output like:

```
Serving functions on http://127.0.0.1:54321/functions/v1/<function-name>
```

Leave this terminal running.

---

## Step 4 — Forward Stripe webhooks to local functions

In a second terminal:

```bash
stripe listen --forward-to http://127.0.0.1:54321/functions/v1/stripe-webhook
```

This tells Stripe to forward all webhook events from your test account to the
local function. Each event will be validated using the `STRIPE_WEBHOOK_SECRET`
you set in `.env.local`.

Leave this terminal running too.

---

## Step 5 — Start the frontend

In a third terminal:

```bash
cd /DATA/retirement
npm run dev
# Vite starts on http://localhost:5173
```

Your `.env.local` in the project root has `VITE_STRIPE_PRICE_MONTHLY` and
`VITE_STRIPE_PRICE_YEARLY` pointing to test price IDs — these are what get
sent to the `stripe-checkout` function.

---

## Step 6 — Trigger a test checkout

1. Open `http://localhost:5173` and sign in
2. Click **Upgrade** and choose a plan
3. The frontend calls `stripe-checkout` (local) → creates a Stripe checkout session
4. Complete checkout using a test card: `4242 4242 4242 4242`, any future expiry, any CVC
5. Stripe fires webhook events → Stripe CLI forwards them to `stripe-webhook` (local)
6. Local function writes to your Supabase `subscriptions` table
7. The app should show Pro features as unlocked

---

## Test Cards (Stripe test mode)

| Scenario | Card number |
|----------|------------|
| Successful payment | `4242 4242 4242 4242` |
| Payment declined | `4000 0000 0000 0002` |
| Requires 3D Secure | `4000 0025 0000 3155` |
| Insufficient funds | `4000 0000 0000 9995` |

Use any future expiry date (e.g., `12/34`) and any 3-digit CVC.

## Triggering webhook events manually

You can fire test events without going through checkout:

```bash
# Simulate a successful subscription creation
stripe trigger customer.subscription.created

# Simulate a failed payment
stripe trigger invoice.payment_failed

# Simulate subscription cancellation
stripe trigger customer.subscription.deleted
```

The Stripe CLI will forward these to your local `stripe-webhook` function.

## Architecture reminder

```
localhost:5173 (Vite + React)
  → calls stripe-checkout at 127.0.0.1:54321 (supabase functions serve)
  → stripe-checkout creates Stripe session (test mode, sk_test_...)
  → user redirected to Stripe-hosted checkout
  → Stripe fires webhook
  → stripe CLI (stripe listen) forwards to 127.0.0.1:54321/functions/v1/stripe-webhook
  → stripe-webhook validates, writes to Supabase subscriptions table
  → frontend reads subscriptions table via SubscriptionContext
```

## Troubleshooting

**"No such price" error from Stripe**
: The `VITE_STRIPE_PRICE_*` values in `.env.local` are test price IDs. Make
  sure `STRIPE_SECRET_KEY` in `supabase/functions/.env.local` is also a test
  key (`sk_test_...`). Mixing test prices with a live secret key causes this.

**Webhook signature verification failed**
: The `STRIPE_WEBHOOK_SECRET` in `supabase/functions/.env.local` must match
  the secret printed by `stripe listen --print-secret`. Re-run that command and
  update the `.env.local` if you're unsure.

**Functions not picking up env changes**
: Restart `supabase functions serve` after editing `.env.local`.

**Port conflicts**
: If port 54321 is in use, check `lsof -i :54321` and kill the conflicting
  process, or pass `--port <n>` to `supabase functions serve`.
